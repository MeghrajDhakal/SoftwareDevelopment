Chapter 9: User Interface Design

9.1 Basics of User Interface

Definition:

The user interface portion of a software product is responsible for all interactions with the user. Almost every software product has a user interface. Users often judge a software product based on its user interface.

  • An interface that is difficult to use leads to higher user errors and dissatisfaction.
  • Users build a mental model of the system and expect behaviour to conform to it.
  • For many interactive applications, as much as 50% of total development effort is spent on UI.

Users become particularly irritated when a system behaves in unexpected ways — i.e., issued commands do not carry out actions according to their intuitive expectations.

9.2 Characteristics of a Good User Interface

A good UI should possess the following characteristics:

  1. Speed of Learning: Easy to learn. Enhanced by Metaphors (shopping cart, desktop), Consistency (similar commands for similar actions), and Component-based UI.
  2. Speed of Use: Time and effort to initiate commands. Most used commands should be shortest/quickest to access.
  3. Speed of Recall: Fast recall for intermittent users. Intuitive names and symbols help.
  4. Error Prevention: Minimize scope for errors through consistency and validation.
  5. Aesthetic and Attractive: Catches user attention. Graphics-based generally beats text-based.
  6. Consistency: Allow users to generalise knowledge from one part to another.
  7. Feedback: Provide status if processing takes more than a few seconds. Prevents user panic.
  8. Support for Multiple Skill Levels: Novices need step-by-step; experts need hotkeys and macros.
  9. Error Recovery (Undo): Essential so users feel in control and not helpless.
  10. User Guidance: Help system for those unaware of features.
EXAM TIP: Pick the top 5 for a 5-mark question: Learning, Use, Error Prevention, Feedback, Consistency.

9.3 Help System

On-line Help System:

Provides help on system operation anytime. It should be context-sensitive, tailored to experience, user-friendly, and NOT just a copy of the manual.

  • Guidance Messages: Prompt the user about next actions or current status.
  • Error Messages: Should be polite, specific, and constructive (suggesting how to rectify).

9.6 Types of User Interfaces

  • Command Language-based: Fast interaction requiring minimal typing for experts. But difficult to learn and error-prone.
  • Menu-based: Based on recognition rather than recollection. (Scrolling, Walking/Cascading, Hierarchical, Pop-up, Pull-down).
  • Direct Manipulation: Visual models (icons). Highly intuitive, but slow for experts.

9.7 Component-based GUI Development

Every interface is built from predefined components (widgets). This reduces programmer work to integration and ensures consistency.

Visual Programming: Drag-and-drop style development where icons represent GUI components. Example: Visual Basic.

9.11 UI Design Methodology (Frank Ludolph)

  1. Examine the Use Case Model (tasks users perform).
  2. Task and Object Modelling (structure of tasks).
  3. Metaphor Selection (e.g. Shopping Cart, Desktop).
  4. Interaction Design and Rough Layout.
  5. Detailed Presentation and Graphics Design.
  6. GUI Construction (Modal vs non-modal dialogs).
  7. Usability Evaluation (Nielsen's Checklist).
Implications of Human Cognition:

Limited memory (7±2 items). Provide frequent task closures to give user relief. Prioritize recognition over recall.

Chapter 6: Function-Oriented Design (SA/SD & DFD)

6.1 & 6.2 SA/SD METHODOLOGY

Function-Oriented Design:

Views system as a black box. Successively decomposes high-level functions via Top-Down decomposition.

SA (Structured Analysis): Transforms SRS into a DFD (What system does).
SD (Structured Design): Transforms DFD into a Structure Chart (How it's structured).

flowchart LR SRS[SRS Document] -- SA --> DFD[DFD Model] DFD -- SD --> SC[Structure Chart]
Figure 6.1 — SA/SD Methodology

6.4 & 6.5 Data Flow Diagrams (DFDs)

DFD is a hierarchical bubble chart displaying processing activities and data flow. It absolutely DOES NOT verify control flow, algorithmic logic or loops.

DFD Symbols

SymbolNameDescription
CircleFunction / BubbleAnnotated with a Verb (except context level).
RectangleExternal EntitySource/destination of info.
Parallel LinesData StoreLogical or physical file/database.
ArrowData FlowMovement of data.

Synchronous: Bubbles directly connected by flow. Asynchronous: Connected via Data Store.

Data Dictionary: Single repository listing all data items across all levels.

6.7 Developing the DFD Model

  • Level 0 (Context): Only ONE bubble (Noun). Shows all external boundaries.
  • Level 1: 3-7 bubbles (Verbs). External entities removed/hidden.
  • Balancing: Data flowing into/out of a bubble must exactly match the inflows/outflows of its next-level decomposition.

Chapter 7: Object Modelling Using UML

7.1 OOP Concepts

  • Class & Object: Class is ADT template, Object is the instance.
  • Abstraction: Hiding irrelevant details.
  • Encapsulation: Data hidden inside methods; weak coupling.
  • Polymorphism: Static (compile-time overloading) and Dynamic (run-time overriding).
  • Inheritance: Superclass generic logic reused in subclass.
